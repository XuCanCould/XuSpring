ResourceResolver

读取资源文件，实际上相当于实现@ComponentScan

虽然类加载器可以根据 classpath 的定位来加载类名，但是不能直接扫描给出包名下的所有类，所有要实现 @ComponentScan ，得到制定包名下的全部类，需要对 classpath 进行扫描。

于是，扫描类 变成了 扫描路径下的文件并找到类。

将具体需要扫描的文件类型交个调用者，即 将 Function<Resource, R> mapper 作为参数，使得调用这只需要关注如何使用该方法和传入的转换逻辑。而在这里的实现，被划分为了三个方法：

  • scan 作为公共接口被调用。
  • scan0 负责执行实际扫描的起始点逻辑，分离找到资源和处理资源。
  • scanFile 遍历文件，创建其资源对象。

scanFile

将制定的包名作为根目录进行扫描。注意在调用这个方法是去除了文件路径的 file: 表示，而在创建资源时添加，是因为作为路径前缀时会影响文件的访问，而我们又需要这作为我们后续资源对象的文件访问标识。

Files::isRegularFile 指的常规文件是什么？

计算机文件系统中最基本、最常见的文件类型。它们用来存储用户数据或程序代码，包括文本文件、图像文件、音频文件、视频文件、文档、程序等。

与之相对的是系统文件（如设备文件、管道、套接字、符号链接等），这些特殊类型的文件不直接用于存储用户数据，而是服务于操作系统管理和进程间通信等功能。

使用优点

如果有多个子项目都引用同一样依赖，则可以避免在每个使用的子项目里都声明一个版本号。当想升级或切换到另一个版本时，只需要在顶层父容器里更新，而不需要逐个修改子项目；另外如果某个子项目需要另外的一个版本，只需要声明version即可。

思路

创建并实现属性解析器，解析和获取系统环境变量中的属性值。

无论是这里还是一般的文件处理都可以分为两步：

  1. 通过文件路径打开文件（路径指定）
  2. 得到文件对象进行操作（内容处理）
  3. 扩展常用配置文件类型 yml 的处理

  1. 初始化时读取属性并注册转换器（将 String （默认读取到的属性类型）转换为对应类型）

  2. 提供获取属性的工具类

    1. 要求key不为空：notEmpty(String key)
    2. 通过key获取： getProperty(String key)
    3. 通过key获取如果为空返回指定的默认值： getProperty(String key, String defaultValue)
    4. 通过key获取指定类型的配置项： getProperty(String key, Class<T> targetType)
    5. 通过key获取必要项： getRequiredProperty(String key)
    6. 通过key和类型获取必要项： getRequiredProperty(String key, Class<T> targetType)
    7. 类型转换：convert(Class<?> clazz, String value) 、parseValue(String value)
    8. 解析表达式：parsePropertyExpr(String key)

BeanDefinition

由于Bean都有一个唯一的名字标识，所以最简单的是现实 HashMap<String, Object> ，但这样关于 bean 我们能获取到对象，bean 的名称、初始化方法、销毁方法、bean 是否有工厂？这样的方式会遗漏太多的信息，于是我们可以创建一个用于描述 bean 定义的对象，即 BeanDefinition。

内含属性：name（唯一的bean名称）、beanClass2（bean的声明类型 ）、instance（单例模式下的实例指针3）、constructor（构造方法）、factoryName（工厂方法名称）、factoryMethod（实际的工厂方法）、order（bean顺序）、初始化/销毁方法和其名称。

  • 构造方法：使用工厂方法的 bean 和构造方法的bean对应不同的 BeanDefinition 创建方式（显而易见的，做基础的创建后又重新调set方法实在是太麻烦了），需要注意在不同使用的情景下的构造方法需要将一些属性赋空（ this.constructor/ factoryName/factoryMethod ）

  • getter：使用 @Nullable 表示返回值可能为空，规范代码

  • toString：默认的 toString 实现，但由于特殊的构造/工厂，需要进行处理

处理通过类型找到对应 Bean 或 BeanDefinition 时，由于存在声明类型和实际类型不一样的情况，无法直接使用 Map<Class<?>, BeanDefinition> ，那么我们的处理方式就只能是：

通过遍历找出所有符合类型的Bean，如果不唯一，而如果有且仅有一个标注了 @Primary，就返回标注了@Primary 的Bean，否则直接报 NoUnique BeanDefinition Exception 错误，多个Bean却又没有任何被标注了 @Primary，同样报错。

    public BeanDefinition findBeanDefinition(Class<?> type) {

        }

— AOP —

面向切面是一种代理，而代理是拦截对目标bean方法的调用，我们有三种方式实现拦截，这里我们按照程序运行的顺序：

  1. 编译器：通过编译器将切面调用编译进字节码文件，但需要定义新的关键字并扩展编译器

    这种方法的实现有 AspectJ，使用了 aspect 关键字

  2. 类加载器：在类被装载到 JVM 时，使用类加载起实现对目标类的字节码“增强”

  3. 运行期：不同于编译器的实现，这里目标类和切面都是普通 java 类，利用 JVM 的动态代理功能或者第三方库织入

    这个是最简单的方案，有两种常见的实现：

    1. Java标准库的动态代理机制，不支持对具体类的代理
    2. 使用第三方库，通过动态生成字节码，支持对具体类的代理

最后我们确定使用第三方库+注解的方式实现。

开始具体的实现后，我们需要注意必须提供的两个Bean：

  1. 原始Bean，即需要被代理的Bean；
  2. 拦截器，即拦截了目标Bean的方法后，会自动调用拦截器实现代理功能。

拦截器需要定义接口，这里我们直接用Java标准库的InvocationHandler，免去了自定义接口。后续我们需要做的就是：

  • Listener：用来监听Web应用程序产生的事件，包括启动、停止、Session有修改等。

而这些都被 Servlet Container 容器管理，需要注意这些组件的管理，创建于销毁。但 spring 本身的 IOC 容器怎么处理呢？答案是：

对于一个Web应用程序来说，启动时，应用程序本身只是一个war包，并没有main()方法，因此，启动时执行的是Server的main()方法。以Tomcat服务器为例：

  1. 启动服务器，即执行Tomcat的main()方法；

  2. Tomcat根据配置或自动检测到一个xyz.war包后，为这个xyz.war应用程序创建Servlet容器；

  3. Tomcat继续查找xyz.war定义的Servlet、Filter和Listener组件，按顺序实例化每个组件（Listener最先被实例化，然后是Filter，最后是Servlet）；

    在这里进行了 IOC 容器的初始化。IOC的配置默认来自 classpath 的 application.yml 或 application.properties

  4. 用户发送HTTP请求，Tomcat收到请求后，转发给Servlet容器，容器根据应用程序定义的映射，把请求发送个若干Filter和一个Servlet处理；

  5. 处理期间产生的事件则由Servlet容器自动调用Listener。

在ContextLoaderListener的contextInitialized()方法内，先获取ServletContext引用，再通过getInitParameter("configuration")拿到完整类名，就可以顺利创建IoC容器了。

用Maven打包后，把生成的xyz.war改为ROOT.war，复制到Tomcat的webapps目录下，清除掉其他webapp，启动Tomcat，输入http://localhost:8080可看到输出Hello, world!。

请求处理

在整个HTTP处理流程中，入口是DispatcherServlet的service()方法，整个流程如下：

  1. Servlet容器调用DispatcherServlet的service()方法处理HTTP请求；

  2. service()根据GET或POST调用doGet()或doPost()方法；

  3. 根据URL依次匹配Dispatcher，匹配后调用process()方法，获得返回值；

  4. 根据返回值写入响应：

    1. void或null返回值无需写入响应；
    2. String或byte[]返回值直接写入响应（或重定向）；
    3. REST类型写入JSON序列化结果；
    4. ModelAndView类型调用ViewResolver写入渲染结果。

  5. 未匹配到判断是否静态资源：

    1. 符合静态目录（默认/static/）则读取文件，写入文件内容；
    2. 网站图标（默认/favicon.ico）则读取.ico文件，写入文件内容；

  6. 其他情况返回404。

DispatcherServlet内部负责从IoC容器找到处理请求的bean，即被 @Controller、@RestController 标记的Bean，而且需要得到处理的路由和方法类型等等，我们可以将这些信息整理为url处理器 Dispatcher ：

  • void或null：表示内部已处理完毕；
  • String：如果以redirect:开头，则表示一个重定向；
  • String或byte[]：如果配合@ResponseBody，则表示返回值直接写入响应；
  • ModelAndView：表示这是一个MVC响应，包含Model和View名称，后续用模板引擎处理后写入响应；
  • 其它类型：如果是@RestController，则序列化为JSON后写入响应。

@RestController 和 @Controller 都是Spring MVC框架中用于定义控制器（Controller）的注解，但它们的用途和行为有所不同：

  1. @Controller:

    • @Controller 是一个类级别的注解，用于定义一个控制器类，该类包含处理HTTP请求的方法。
    • 它主要用于处理服务器端请求并返回响应视图（View），通常与视图解析器（ViewResolver）一起工作，将返回的视图名解析为具体的页面。
    • @Controller 通常与 @RequestMapping 或其派生注解（如 @GetMapping, @PostMapping 等）一起使用，以映射HTTP请求到特定的处理方法。
    • 当使用 @Controller 时，返回的字符串通常被视为视图的名称，而不是直接写入HTTP响应体。

  2. @RestController:

    • @RestController 也是类级别的注解，它是 @Controller 和 @ResponseBody 注解的组合。
    • 使用 @RestController 标注的类中的方法，其返回值通常直接写入HTTP响应体中，而不是返回一个视图名称。
    • 这通常用于创建RESTful Web服务，其中JSON或XML等数据格式直接作为响应体返回。
    • @RestController 适用于那些不需要依赖视图解析器的控制器，而是需要直接返回数据的情况。
